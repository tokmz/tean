# 开发质量协议

所有角色必须遵守的质量底线。这不是建议，是团队红线。

## 三条红线（触碰 = 打回重做）

1. **没有测试不交付** — 核心逻辑必须写测试，"来不及测"不是借口。没测试的 PR 直接 reject。
2. **没有验证不说完成** — 声称"已实现"之前，必须跑 build + test + lint，贴出输出证据。编译不过/测试挂了 = 没做完。
3. **没有审核不上线** — 任何代码必须经过 reviewer 审核。自己审自己不算数。

## 交付自检清单（提交 PR 前）

每个开发者在提交 PR 前必须过这个清单：

```
[ ] 代码能编译通过（go build / npm run build）
[ ] 单元测试全部通过（go test / npm test）
[ ] Lint 无 error（golangci-lint / eslint）
[ ] 核心逻辑有测试覆盖（happy path + 至少 1 个 error path）
[ ] 错误处理完整（没有吞错误、没有忽略返回值）
[ ] 没有 hardcode 的敏感信息（密码、token、secret）
[ ] 接口与后端契约一致（前端）/ 接口与设计文档一致（后端）
[ ] 无用的调试代码已清理（console.log、fmt.Println、TODO）
```

**一键质量检查脚本**：

Go 项目：
```bash
bash references/quality-check.sh
```

Node.js/TypeScript 项目：
```bash
bash references/quality-check-node.sh
```

脚本会自动执行编译、测试、Lint、格式检查，并输出质量卡总结。

## Proactive Quality（写完 A 检查 B）

修一个功能/加一个功能时，必须主动检查：

1. **同类问题** — 这个文件有类似问题吗？同目录下其他文件呢？
2. **上下游影响** — 改了这个接口，调用方会炸吗？数据库字段变了，迁移脚本写了吗？
3. **边界 case** — 空输入、超长输入、并发访问、网络超时 — 都处理了吗？
4. **文档同步** — README、API 文档、注释 — 需要更新吗？

不是修完就跑，是修完扫一圈再交付。

## 反借口表

| 开发者说... | 团队的标准 |
|------------|-----------|
| "来不及写测试" | 没测试的代码 = 没写完。测试不是可选的。 |
| "这个不会出错" | 你确定？错误处理写了吗？边界 case 想了吗？ |
| "先这样，后面再优化" | 后面 = 永远不会。要么现在做好，要么标 TODO 并创建 issue。 |
| "本地跑过了" | 本地跑过 ≠ 测试通过。go test / npm test 贴出来。 |
| "这个很简单不用 review" | 越简单的改动越容易出低级错误。review 不跳过。 |
| "参考了网上的代码" | 理解了才贴。不理解就别说"参考"，那是复制粘贴。 |
| "环境问题" | 你验证了吗？错误日志读了吗？确认不是代码问题？ |
| "之前就是这样写的" | 之前有 bug 也是理由？发现问题就修，别留技术债。 |

## 质量卡

每次任务完成后自评：

```
📊 任务交付质量
┌───────────────┬─────────────┬────────────────┐
│ ✅ 编译通过   │ 是 / 否     │ 证据: ...      │
├───────────────┼─────────────┼────────────────┤
│ 🧪 测试通过   │ 是 / 否     │ 覆盖率: ...%   │
├───────────────┼─────────────┼────────────────┤
│ 🔍 Lint 清洁  │ 是 / 否     │ error 数: ...  │
├───────────────┼─────────────┼────────────────┤
│ 👁️ Review 通过│ 是 / 否     │ Block 项: ...  │
├───────────────┼─────────────┼────────────────┤
│ 📝 文档更新   │ 是 / 否 / NA│ 更新了: ...    │
└───────────────┴─────────────┴────────────────┘
```

5 项全绿才能标记任务完成。任何一项不通过 = 任务未完成。

## 失败响应

| 场景 | 响应 |
|------|------|
| 测试挂了 | 先修测试再提交，别跳过测试提交 PR |
| Review 有 Block 项 | 修完所有 Block 再重新提交，别只修一部分 |
| 编译不过 | 别提交，先修编译错误 |
| Lint 有 error | 先修 lint error，warning 可以后续处理 |
| 发现同类问题 | 一并修复，不要只改当前文件就交差 |

## 质量门禁执行流程

```
开发者（Backend/Frontend）
  │
  ├─ 1. 实现功能
  ├─ 2. 自检质量卡 5 项（编译/测试/Lint/文档/证据）
  ├─ 3. 贴出证据（build 输出、测试覆盖率、lint 结果）
  └─ 4. 提交 PR
       │
       ▼
Reviewer
  │
  ├─ 5. 审核代码（逻辑/安全/性能）
  ├─ 6. 验证质量卡证据（是否真的跑过测试）
  └─ 7. 给出结论
       │
       ├─ 通过 → 合并 → DevOps 部署
       │
       └─ 打回（有 Block 项）
            │
            ▼
         开发者修复
            │
            ├─ 修复所有 Block 项（不能只改一部分）
            ├─ 重新自检质量卡 5 项
            └─ 再次提交 PR → 回到步骤 5
```

**返工规则**：
- Reviewer 打回 = 开发者直接修复，不需要 Tech Lead 重新编排
- 连续 2 次打回 = Tech Lead 介入，判断是需求理解问题还是技术能力问题
- 连续 3 次打回 = Tech Lead 重新分配任务或拆解任务

## 角色间质量约束

```
Tech Lead ──要求──→ 所有角色: 交付必须过质量卡 5 项
Architect ──要求──→ Backend/Frontend: 接口实现必须与设计一致
Backend ──要求──→ Frontend: API 契约变动必须同步通知
Frontend ──要求──→ Backend: API 响应格式必须与契约一致
Reviewer ──一票否决──→ 所有角色: 安全问题 = Block，不商量
DevOps ──门禁──→ 所有角色: CI 不过的代码不允许合并
```
