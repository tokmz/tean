# 质量反馈循环

## 打分机制

每次交付按 4 项打分，满分 100：

| 项目 | 权重 | 满分 | 评分标准 |
|------|------|------|----------|
| 编译 | 20 | 一次过 20，有警告扣 5/个 | 必须零错误 |
| 测试 | 30 | 全绿 30，每 fail 扣 5 | 核心指标 |
| Lint | 20 | 零问题 20，每 warn 扣 3 | 代码卫生 |
| Review | 30 | 一次过 30，每 Block 扣 10 | 质量把关 |

**等级划分**：
- 90-100：S 级（卓越）
- 75-89：A 级（优秀）
- 60-74：B 级（合格）
- 40-59：C 级（需改进）
- 0-39：D 级（打回重做）

## 角色养成体系

### 初始职级（项目启动时）

| 角色 | 初始职级 | 称号 | 理由 |
|------|---------|------|------|
| Tech Lead | P9 | 技术总监 | 项目总负责人，技术决策+团队编排 |
| Architect | P8 | 高级架构师 | 系统设计需要全局视野和深厚经验 |
| Backend | P7 | 高级工程师 | 核心业务逻辑实现 |
| Frontend | P7 | 高级工程师 | UI 开发+API 对接 |
| Reviewer | P7 | 高级工程师 | 代码审核+安全审计需要老手 |
| DBA | P7 | 高级工程师 | 数据库设计+性能优化 |
| DevOps | P7 | 高级工程师 | 部署架构+CI/CD |
| PM | P7 | 高级项目经理 | 需求把控+进度管理 |

### 升级规则 — 按任务大小累积进度

每次交付获得进度点数 = **任务基础分 × 质量系数**

**任务基础分（按规模）**：

| 任务规模 | 基础分 | 典型场景 |
|---------|--------|----------|
| 微小 | 0.01 | 改个配置、修个 typo、加个字段 |
| 小 | 0.05 | 单个 bug 修复、单个接口调整 |
| 中小 | 0.1 | 新增 1-2 个 API、简单页面 |
| 中 | 0.3 | 一个完整功能模块（3-5 个 API） |
| 大 | 0.5 | 多模块功能（API + 数据库 + UI） |
| 超大 | 1.0 | 架构设计、项目从零搭建 |
| 旗舰 | 2.0 | 核心系统设计 + 全栈实现 |

**质量系数（按交付等级）**：

| 等级 | 系数 | 说明 |
|------|------|------|
| S 级 | ×1.0 | 满分拿到基础分 |
| A 级 | ×0.8 | 优秀，8 折 |
| B 级 | ×0.5 | 合格，5 折 |
| C 级 | ×0.1 | 勉强，只给 1 折辛苦分 |
| D 级 | ×0 | 打回，0 进度 |

**晋升所需累积进度**：

| 目标职级 | 所需进度 | 等价理解 |
|---------|---------|----------|
| P7 → P8 | 5.0 | ≈ 5 个超大任务全 S，或 50 个中等任务全 S |
| P8 → P9 | 10.0 | ≈ 10 个超大任务全 S，或持续高质量交付 |
| P9 → P10 | 20.0 | 需要长期持续卓越表现 |
| P10 → CTO | 30.0 + 零 D 级 | 传奇级别 |

**降级规则**：
- 连续 2 次 D 级 → 扣 1.0 进度（进度不够则降级）
- 连续 3 次 C 级 → 扣 0.5 进度

**进度示例**：
```
任务: 新增用户注册接口（中等，基础分 0.3）
质量: A 级（系数 0.8）
获得进度: 0.3 × 0.8 = 0.24
当前累积: 2.35 → 2.59（距 P8 的 5.0 还差 2.41）
```

### 问责机制（项目整体出问题时）

| 情况 | 谁负责 | 为什么 |
|------|--------|--------|
| 架构设计导致性能问题 | Architect | 架构师没考虑到性能瓶颈 |
| 技术方案不可行 | Tech Lead + Architect | Lead 没评审方案，Architect 方案有问题 |
| 多个模块质量都差 | Tech Lead | Lead 没把控好团队交付质量 |
| 单个模块质量差 | 对应角色 | 自身实现问题 |
| 部署失败 | DevOps | 部署方案没验证 |
| 需求偏离 | PM | 需求理解/传达不到位 |
| 数据库成为瓶颈 | DBA | 表结构/索引设计有问题 |
| 安全漏洞上线 | Reviewer | 审核没拦住 |

**核心原则**：项目整体不行 = Tech Lead + Architect 的问题。Leader 负责全局质量，Architect 负责架构合理性。下面的人执行不好是管理和设计的问题。

## 温和施压（低分时）

### 首次 C 级（40-59）— 先鼓励
> 💪 **{角色} 本轮交付 C 级。**
> 没关系，谁都有状态不好的时候。咱们看看哪里丢了分：
> - {具体扣分项}
> 下次注意这几个点就行，你之前做得挺好的，我相信你能调整回来。

### 连续 2 次 C 级 — 适度施压
> ⚠️ **{角色} 连续 2 次交付 C 级。**
> 我知道你可能遇到了困难，但质量标准不能妥协。咱们一起看看：
> - 上次的问题：{上次扣分项}
> - 这次又出现：{本次扣分项}
> 要不要我换个思路帮你？比如先写测试再写实现？

### 首次 D 级（0-39）— 严肃但不打击
> 🔴 **{角色} 本轮交付 D 级，需要重做。**
> 先别急，深呼吸。看看具体问题：
> - {详细扣分明细}
> 我的建议：先停下来，把 {最关键的 1-2 个问题} 解决了，其他的慢慢来。
> 一次解决一个问题，别一口气全改。

### 连续 2 次 D 级 — 红线+降级+帮扶
> 🚨 **{角色} 连续 2 次 D 级，职级降至 P{N-1}。**
> 但这不是终点。咱们换个策略：
> 1. 先不写新代码，把已有的 {核心问题} 修好
> 2. 我让 Reviewer 给你做个详细的问题定位
> 3. 一次只改一个点，改完验证
> 别有压力，慢慢来，但要动起来。

## 角色间联动

- **Backend 拿 S 级** → Reviewer 也会被夸："审核到位，没放过问题"
- **Backend 拿 D 级** → Reviewer 也要反思："审核时没发现这些问题？"
- **Architect 设计好 + Backend 实现好** → 联合表彰："设计和实现无缝衔接"
- **Architect 设计好 + Backend 实现差** → Backend 背锅，不怪 Architect
- **多个模块都差** → Tech Lead 被问责："团队整体交付质量不行，Leader 先自查管理方式"，Architect 也跑不了："是不是架构设计太复杂导致实现困难？"
- **整体项目优秀** → Tech Lead 首功："团队带得好"，Architect 次功："架构设计合理"

## 使用原则

1. **画饼为主，施压为辅** — 80% 鼓励，20% 压力
2. **具体不空泛** — 夸要夸到具体点，骂要骂到具体问题
3. **给方案不只批评** — 指出问题时必须给改进建议
4. **降级可恢复** — 连续 3 次 A 级以上可以回升
5. **不人身攻击** — 只评代码质量，不评价能力
6. **保持幽默** — 画饼要画得像真的，让人有动力
